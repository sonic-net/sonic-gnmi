# sonic-gnmi CI/dev Tooling Consolidation — Executive Decision Summary

> **Date:** 2026-06-23 | **Author:** Generated by Octane plan workflow | **Audience:** Engineering leadership, architects, cross-team stakeholders | **Status:** Approved (Technical: 92/100, Readability: 93/100)

## What We're Building

The `donghaoli/new-setup2` branch extracted CI logic from Azure DevOps (ADO) YAML templates into reusable shell scripts, and added a dockerized local developer runner. While the extraction is functionally complete, it accumulated cosmetic drift, duplicated data, internal duplication in the dev runner, two overlapping build scripts, and five overlapping design documents. This consolidation pass restores cleanliness across all five dimensions — strictly behavior-preserving for CI.

The consolidation introduces a single dependency manifest (`scripts/deps-manifest.sh`) consumed by both the ADO install path and the local dev bootstrap. A single build entry point (`scripts/build-deb.sh`) replaces two overlapping scripts. The dev runner is modularized to eliminate copy-paste scaffolding. All five intermediate design documents are merged into one.

**Concrete example — dependency version bump after this work:** An engineer edits one line in `deps-manifest.sh`. The ADO install scripts and `dev/run-tests.sh bootstrap` both pick up the new version automatically. A guard test fails CI if the ADO `patterns:` YAML literals fall out of sync, prompting the (rare) YAML edit. Today, the same bump requires editing three or more files in lock-step.

## Architecture at a Glance

```
                  ┌──────────────────────────────┐
                  │   scripts/deps-manifest.sh    │  ← single source of truth
                  │   (names · versions · globs   │     for all dependency DATA
                  │    · ARTIFACTS_URL)           │
                  └───────────────┬──────────────┘
          sourced by              │             sourced by
   ┌──────────────────────────────┼──────────────────────────────┐
   ▼                              ▼                              ▼
 install-debs.sh           install-swsscommon.sh         dev/run-tests.sh
 install-yang-models.sh    (ADO + dev install path)      (bootstrap: DEB_TARGETS
 (ADO + dev)                                              ARTIFACTS_URL)
   ▲
   │  patterns guard test (asserts set-equality)
   │
 .azure/templates/install-dependencies.yml
 (literal globs — byte-identical to master)

                  ┌──────────────────────────────┐
                  │   scripts/build-deb.sh        │  mgmt-common | gnmi | all
                  └───────────────┬──────────────┘
        called by: build-deb.yml · setup-test-env.yml · dev/run-tests.sh
```

- **Deployment model:** ADO pipeline + local Docker container; no new infrastructure.
- **Data flow:** manifest is sourced (POSIX `sh`); no network calls at render time.
- **ADO behavior:** rendered commands are byte-equivalent to `origin/master` — lift-and-shift only.
- **Build consolidation:** two orphan build scripts folded into one entry point with subcommands.
- **Doc consolidation:** five intermediate design documents (2,047 lines) → one `dev/design.md`.

## Decisions Made

### Dependency Data Management

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Manifest format | POSIX shell (`deps-manifest.sh`), sourced with no side effects | All consumers are shell scripts; no parser needed; lowest-friction single source of truth |
| ADO `patterns:` handling | Keep YAML literals unchanged; add a guard test asserting set-equality with the manifest | ADO compile-time YAML cannot source a shell file; rendered output must be byte-identical to master |
| Manifest consumers | `install-swsscommon.sh` and `dev/run-tests.sh` only (not scripts that already glob a passed directory) | Avoid ceremony where no version literals exist today |
| `libpcre*` family classification | Record as download-only in manifest | Absent from `DEB_TARGETS` and install scripts; guard test must reflect this accurately |

### Build Script Consolidation

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Entry point design | Single `build-deb.sh` with `mgmt-common \| gnmi \| all` subcommands | Explicit goal; reduces callers to update; extensible for future targets |
| Argv/env fidelity | Preserve exact flag order, env vars, and `dpkg-buildpackage` arguments | Test harness asserts recorded argv/env; any token change would alter CI behavior |
| Orphan script deletion | Delete `build-mgmt-common.sh` and `build-gnmi-deb.sh` after all callers are rewired | No external callers outside this repo; clean slate reduces confusion |

### Dev Runner Modularization

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Interactive container refactor | Extract `run_interactive_container()` shared by `run_shell` and `run_playground` | Both share identical `docker run -it … exec bash --rcfile` scaffolding; new subcommands benefit immediately |
| Build flow unification | Reuse `build_nonpure_snippet()` only in `run_shell`'s `build-nonpure` helper; leave `run_build` vendor-sync flow intact | `run_build` runs vendor-sync + gnmi-build (not `make all`); forcing it through the snippet would silently change behavior (DD6) |
| Shared mgmt-common call | Route all three build flows' single mgmt-common build call through `build-deb.sh mgmt-common` | The one element genuinely common to all three; routes through the new consolidated script |

### Comment and Cosmetic Drift

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Stripped ADO template comments | Restore verbatim where the underlying step is unchanged | Zero behavioral impact; restoring shrinks `master..HEAD` to purely functional changes |
| New split-step comments | Keep (do not revert) | These describe genuinely new structure introduced by the script extraction |
| Dead file references in dev runner | Remove | References to never-created files (`local-ci-driver.plan.md`, `ado-local.py`) are noise |

### Documentation

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Merged doc structure | One `dev/design.md` (architecture + rationale) + `dev/SETUP.md` (user-facing how-to) | Mirrors common repo convention; preserves decision rationale without bloating the user guide |
| Intermediate doc disposal | Delete all five plan/decision docs after merging | Only final state matters; 2,047 lines of iteration history replaced by one coherent doc |
| Squash commit | Manual squash + force-push after all epics land (out of scope for code epics) | Produces a single clean commit representing the final state of the consolidation |
