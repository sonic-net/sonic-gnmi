# Local Dev Runner — Executive Decision Summary

> **Date:** 2026-06-22 | **Author:** Generated by Octane plan workflow | **Audience:** Engineering leadership, architects, cross-team stakeholders | **Status:** Approved (Technical: 92/100, Readability: 94/100)

## What We're Building

The existing `dev/run-tests.sh` script already provides a clean, containerized inner-loop for sonic-gnmi developers (bootstrap, unit tests, integration tests, packaging, interactive shell). This initiative makes the **smallest useful evolution** to that script: enabling focused, single-package integration runs rather than always executing the full test suite, and introducing a new **playground** mode that boots a live gNMI/gNOI telemetry server with client tools for manual exploration — without any build system, pipeline, or source-code changes.

The scope is deliberately narrow. A fuller CI-mirror capability (static analysis gates, memory-leak checks, coverage thresholds, multi-arch packaging) is already designed in a companion document but is explicitly deferred. The result is a single Bash script with clearly marked extension points so that richer capabilities can be added incrementally with minimal churn.

A concrete developer workflow: `./dev/run-tests.sh playground 8080` launches Redis, builds the server and all client tools inside the container, starts `telemetry` in no-TLS/insecure mode, and drops the developer into an interactive shell where they can immediately run `gnmi_cli -a 127.0.0.1:8080 -insecure` or `gnoi_client -target 127.0.0.1:8080 -insecure` against a live service — then exit to tear everything down automatically.

## Architecture at a Glance

```
Developer CLI
   │
   ▼
dev/run-tests.sh  (single Bash script, one case dispatch + one usage function)
   ├── bootstrap      clone siblings + fetch artifacts         [UNCHANGED]
   ├── pure           fast Go unit tests, no SONiC deps        [UNCHANGED]
   ├── integration    full suite OR focused package subset     [EXTENDED]
   │    └── resolve_pkg: short-name → module path + tier
   ├── playground     redis + build + server + interactive shell [NEW]
   ├── build          debian packaging                          [UNCHANGED]
   ├── shell          interactive container shell               [UNCHANGED]
   ├── all            bootstrap → pure → integration            [UNCHANGED]
   └── clean          wipe cache                                [UNCHANGED]
          │
          ▼  docker run (sonic-slave-trixie)
          │   bind mounts: sonic-gnmi, sonic-mgmt-common,
          │                sonic-swss-common, sonic-debs (ro)
          ▼
     Makefile targets / telemetry binary  (single source of truth)
```

- **Deployment model:** containerized; all build/test/server logic runs inside the existing `sonic-slave-trixie` image — zero new host dependencies.
- **Data flow:** the script is pure orchestration; Makefiles own all build and test logic; the `telemetry` binary owns server behavior.
- **Subset mechanism:** Makefile variable overrides (`INTEGRATION_*_PKGS`) let the script scope test runs without duplicating build or junit logic.
- **Auth model for playground:** no-TLS / insecure flags built into the server binary; explicitly scoped to ephemeral local containers only.
- **Extensibility:** a labeled comment block names the deferred CI-parity subcommands so future contributors know exactly where to add them.

## Decisions Made

### Script Architecture

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Language and structure | Single Bash script, one `case` dispatch, one `usage` function | Matches the existing tool; zero new host dependencies; lowest maintenance burden |
| Scope of changes | Two files only: the driver script and the setup doc | No Makefile, pipeline, proto, or source changes; strictly additive |
| Extension strategy | Labeled comment block naming deferred subcommands in the dispatch section | Makes the growth path explicit without implementing it; reduces future churn to two edits per new subcommand |
| Composite `all` target | Unchanged: bootstrap → pure → integration | Preserves the existing fast-loop semantics; `playground` is never wired into `all` |

### Integration Subset

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Subset mechanism | Pass `INTEGRATION_BASIC_PKGS` / `INTEGRATION_ENV_PKGS` / `INTEGRATION_DIALOUT_PKG` overrides into the existing `check_gotest_junit` target | Reuses existing gotestsum + junit + env handling; one source of truth for test config |
| Tier management | All three package-tier variables are always explicitly overridden when a subset is given (unused tiers set to empty) | Prevents dialout tests from running on unrelated subsets when `ENABLE_DIALOUT_VALUE != 0` |
| Package name resolution | A `resolve_pkg` helper maps short names to full module paths and classifies each into its tier | Single location for the short-name table; DRY; already-qualified paths pass through verbatim |
| Alternative rejected | Hand-rolled `go test` bypassing Makefile | Duplicates CGO/env/junit logic; creates a second source of truth |
| `-run` regex pass-through | Deferred | Achievable later via `TEST_FLAGS`; not needed for current scope |

### Playground

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Build step for server/tools | Reuse existing `make all` (via `build_nonpure_snippet`); no extra `make` invocation | `make all` already installs all client binaries; dedicated targets for individual binaries do not exist |
| TLS / auth mode | `--noTLS --insecure --allow_no_client_auth` | No local CA; server flags are purpose-built for local testing; acceptable because server runs in a throwaway container on the developer's machine only |
| Access surface | TCP port (default 8080, overridable) published to host + Unix domain socket inside container | Supports both `gnmi_cli`/`gnoi_client` over TCP from host and `grpcurl unix://` inside container |
| Session lifecycle | Server runs in background; `exec bash` drops developer to interactive shell; container `--rm` tears everything down on exit | No cleanup step needed; nothing persists |
| Alternative rejected | Running the server on the host | Requires manual dependency installation (swss-common, libyang, redis); negates the containerized approach |
| `playground` and `all` | `playground` is never invoked by `all` or any test target | It is a manual exploration tool, not a test; must remain independent |

### Security

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Plaintext flags scope | Permitted in `playground` only; documented prominently; `build`/deploy path unchanged | Risk is acceptable for an ephemeral local container; must not be copied into production flows |
| Port binding | Published port should bind to localhost; documented in SETUP.md | Limits blast radius to the developer's machine |

## Follow-up Design Items

| Item | Priority | Description |
|------|----------|-------------|
| `playground --uds-only` mode | Low | Option to skip TCP port publishing for developers who only need `grpcurl unix://` access inside the container |
| `-run <regex>` pass-through on `integration` | Low | Allow filtering individual test cases within a package subset; achievable via `TEST_FLAGS` when needed |
| Full CI-parity subcommands | Medium | `staticcheck`, `memleak`, `coverage`, `ci`, `build --arch` — fully designed in `dev/local-ci-driver.plan.md`; extension points are pre-marked in the script |
