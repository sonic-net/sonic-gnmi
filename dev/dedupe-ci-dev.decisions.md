# CI ↔ Dev Build Logic Deduplication — Executive Decision Summary

> **Date:** 2026-06-22 | **Author:** Generated by Octane plan workflow | **Audience:** Engineering leadership, architects, cross-team stakeholders | **Status:** Approved (Technical: 92/100, Readability: 93/100)

## What We're Building

The same setup, build, and test-prep shell logic is currently maintained twice: once inside the local developer driver (`dev/run-tests.sh`) and once inside Azure DevOps (ADO) pipeline YAML templates. Five concrete overlaps exist — redis socket configuration, sonic-mgmt-common package build, sonic-gnmi `.deb` build, SONiC dependency installation, and test entry points. When either copy drifts, developers get the silent failure mode this tooling was designed to prevent: "passes locally, fails in CI" (or vice-versa).

This initiative extracts the environment-agnostic shell bodies into small, single-purpose scripts under a new repo-root `scripts/` directory. Both the ADO templates and the dev driver become thin callers of these shared scripts, parameterizing only the genuine differences (working directory paths, output destinations). Test entry points are explicitly excluded because the Makefile already serves as their single source of truth.

A concrete example: after this change, the entire redis test configuration lives only in `scripts/setup-redis.sh`. The ADO template calls `sonic-gnmi/scripts/setup-redis.sh`; the dev driver calls `bash /work/sonic-gnmi/scripts/setup-redis.sh`. A maintainer who renames the redis socket changes exactly one file, and both environments are updated atomically.

## Architecture at a Glance

```
         repo-root scripts/   ← SINGLE SOURCE OF TRUTH
         ├── setup-redis.sh          (redis socket config)
         ├── build-mgmt-common.sh    (NO_TEST_BINS=1 deb build)
         ├── build-gnmi-deb.sh       (ENABLE_TRANSLIB_WRITE deb build)
         └── install-sonic-deps.sh   (dep install verbs — optional)
                    ▲                          ▲
    sonic-gnmi/scripts/x.sh        bash /work/sonic-gnmi/scripts/x.sh
                    │                          │
     .azure/templates/*.yml            dev/run-tests.sh
   (keeps Download/checkout/          (keeps docker mounts,
    publish as native ADO)             local cache, chown,
                                       go mod vendor)
```

- **Sharing boundary:** only environment-agnostic shell bodies are extracted; ADO-native tasks (`DownloadPipelineArtifact@2`, `- checkout:`) and dev-native mechanics (bind mounts, local cache) remain with their respective callers.
- **Path discipline:** ADO multi-repo checkout places the repo in a subdirectory, so all ADO invocations use the `sonic-gnmi/`-prefixed path; dev invocations use the container-mounted absolute path.
- **Parameterization model:** differences (directory, output path, artifact glob) are explicit positional arguments — scripts never branch on environment detection.
- **No new dependencies:** scripts use only tools already present in both environments (`sed`, `dpkg`, `apt-get`, `pip3`, `dpkg-buildpackage`).
- **Composability:** the `scripts/` pattern is a deliberate prerequisite for the deferred full CI-mirror plan (`local-ci-driver.plan.md`), which will add new subcommands as additional `scripts/*.sh` rather than re-inlining.

## Decisions Made

### Sharing Strategy

| Decision | Choice | Rationale |
|----------|--------|-----------|
| What to share | Shell body only; not orchestration | ADO-native tasks and dev docker mechanics have no portable equivalent; forcing them into scripts would require environment-detection branching |
| Script granularity | One script per logical unit (redis, mgmt-common build, gnmi build, dep install) | A monolith script cannot be used piecemeal by ADO jobs that have different step layouts (build-deb has no redis; install-dependencies has no gnmi build) |
| Script location | Repo-root `scripts/` | Both consumers invoke these scripts; placing them under `.azure/` would mislabel shared assets as CI-only |
| Environment differences | Expressed as positional arguments with sensible defaults | Keeps scripts trivially testable, prevents environment-specific code paths, makes intent explicit |
| Test entry points (Overlap 5) | Not extracted | Already single-implementation in the Makefile; wrapping adds indirection without removing duplication |

### Behavioral Changes

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `--no-install-recommends` for redis install | Standardize on flag (accepted change for ADO path) | Dev driver already used the flag; ADO did not — standardizing is a deliberate, benign change for a test-only redis install |
| `apt-get update` placement | Stays in ADO caller, not in the script | Required for the script's own `apt-get install redis-server`; dev container refreshes its index earlier, so including it in the script would be redundant on the dev path |
| `-j$(nproc)` for gnmi deb build | Standardize on parallel build (accepted speedup for dev path) | ADO already uses `-j$(nproc)`; dev did not — standardizing adds a small build speedup; no functional difference |
| Artifact glob for gnmi deb | ADO defaults to `*.deb` (copies mgmt-common + gnmi debs); dev caller passes `sonic-gnmi_*.deb` | Preserves each environment's existing published/staged artifact set exactly |

### Rollout Sequencing

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Phase order | Redis → Builds → Dep install (optional) → Sync guard | Start with lowest-risk units; validate the path-resolution pattern in CI before touching the riskier dependency install |
| Epic 3 (dep install) conditionality | Proceed only if ADO caller can adopt without restructuring downloads | ADO never co-locates all debs+wheel in one directory; a one-caller "shared" script delivers zero dedup — drop rather than add |
| Sync guard (Epic 4) | Lightweight CI grep-guard on tell-tale inlined patterns | Cheap insurance against future re-duplication; detects the exact failure mode that motivated this work |

### Rejected Alternatives

| Decision | Rejected Choice | Rationale |
|----------|-----------------|-----------|
| Script structure | Single monolithic setup script | ADO interleaves artifact-download tasks between steps; a monolith would require flags to skip sub-parts |
| Script location | `.azure/scripts/` | Mislabels shared assets as CI-only; dev driver referencing `.azure/…` is semantically misleading |
| Implementation approach | YAML codegen from Bash | Heavy machinery, new build step, obscures YAML; overkill for ~15 lines of shared shell |
| Implementation approach | Status quo with review discipline | Already drifts; this is the problem being solved |
